# 腾讯 CodeBuddy 使用教程：从上手到高效使用的完整实践

## 教程介绍

这两年 AI 编程工具的变化特别快，早就不是“补个代码提示”那么简单了。现在真正好用的工具，核心竞争力已经变成：能不能理解整个项目，能不能跨文件处理任务，能不能自己拆步骤执行，以及能不能把本地文件、任务上下文和工作流真正串起来。

腾讯 CodeBuddy 现在走的就是这条路。无论你把它看成代码助手、桌面智能体，还是一个既能处理开发任务又能处理办公任务的 AI 工作台，它的重点都不只是“回答问题”，而是“真正执行任务并交付结果”。

![CodeBuddy 简介页面截图](/posts-img/codebuddy-overview-shot.jpg)

这篇文章会按实战顺序来讲，主要覆盖这些内容：

- 腾讯 CodeBuddy 是什么，适合怎么用
- 新手第一次上手怎么开始
- CodeBuddy 的工作模式、任务创建和结果查看怎么配合
- 如何通过 MCP 和配置文件把能力扩展起来
- 怎么把 CodeBuddy 真正用进日常开发和办公流程

文章参考了腾讯官方文档、CloudBase 文档，以及几篇技术社区里的实操文章。我尽量把“官方说法”和“真正动手时会遇到的问题”揉在一起，不搞纸上谈兵。

## 腾讯 CodeBuddy 是什么

根据 WorkBuddy / CodeBuddy 官方文档的介绍，WorkBuddy 是腾讯推出的全场景职场 AI 智能体桌面工作台，而 CodeBuddy 则可以理解成其中面向编程和技术协作更强的一条产品线。它支持自然语言理解、自主规划执行、多任务并行、本地文件操作，以及文档、表格、PPT、代码、数据分析等多种任务类型。

这个定位有两个很明显的特点。

第一，它不是单纯的聊天机器人。传统 AI 对话更像“你问一句，它回一句”，而 CodeBuddy 更强调“你说一个目标，它自己拆解并执行”。

第二，它不是只面向开发者。官方文档里，工作模式覆盖了文档生成、数据分析、报告制作、批量文件处理这些办公场景；编程模式则覆盖代码生成、代码审查、Bug 修复、重构优化和项目理解。所以它并不是一个狭义的 IDE 插件，而是一个既能写代码，也能处理实际工作流的智能体平台。

---

## CodeBuddy 和普通 AI 对话工具的区别

官方文档里有一组对比特别直白：传统 AI 往往只能给建议，而 WorkBuddy / CodeBuddy 可以直接执行任务；传统 AI 输出的是一段文字，而 CodeBuddy 更强调“交付可验收的结果”；传统 AI 处理的是单步骤问题，而 CodeBuddy 针对的是多步骤复杂任务。

如果翻译成人话，大概就是：

- 不是只会说，而是真的能干
- 不是只能看你贴出来的内容，而是能直接读本地文件和项目目录
- 不是只能帮你想，而是能帮你把流程跑完

![CodeBuddy 快速开始页面截图](/posts-img/codebuddy-quickstart-shot.jpg)

对开发者来说，这意味着 CodeBuddy 不只是“帮你补一行函数”，而是可以在理解仓库上下文后完成一个完整任务。对非开发场景来说，它也不只是“帮你想 PPT 结构”，而是真的可以在工作目录里生成文档、表格、演示稿或者分析结果。

## CodeBuddy 适合什么场景

结合官方快速开始和创建任务文档，CodeBuddy 大概适合下面几类任务。

### 1. 项目理解和技术开发

比如你接手一个陌生仓库，想快速知道目录结构、启动命令、核心模块和数据流。这类任务如果自己啃，经常要在 README、package.json、脚本文件和业务代码之间来回跳，挺费脑子。CodeBuddy 这类工具的价值就是帮你先梳理一遍，再根据目标继续动手。

### 2. 代码修复和重构

像修一个 Bug、统一接口风格、把某个功能模块改成更合理的结构，这类任务比较适合它。尤其是已经知道问题范围，但不想手工满项目搜改动时，体验会明显好很多。

### 3. 文档和技术资料输出

它不仅能写技术文档、README、方案说明，也能结合目录和现有文件生成更贴近实际项目的内容。这种能力对团队交接、文档补齐和方案沉淀很有帮助。

### 4. 数据分析和自动化处理

官方文档明确提到可以分析上传的数据、自动生成图表，或者批量处理文件。也就是说，CodeBuddy 并不局限于“代码”，它更像一个以任务为中心的执行型助手。

### 5. 长流程任务协作

当一个任务不是一步就能完成，而是需要先理解、再规划、再生成结果时，CodeBuddy 的优势会更明显。它更适合处理“有上下文、有目标、有交付物”的任务，而不是只回答一个零散问题。

## 第一次上手 CodeBuddy，怎么开始

官方快速开始文档给的路线其实很简单：登录、进入主界面、在新建任务页输入一句话任务、选择模型、发送，然后让它开始执行。

### 第一步：进入主界面

打开 WorkBuddy / CodeBuddy 后，如果还没登录，会先看到登录引导。登录完成后，会进入新建任务页面。

### 第二步：直接描述任务

你不需要先学很多命令。官方强调它的核心优势就是理解自然语言，所以最推荐的方式是直接说目标，例如：

- 帮我梳理这个项目的启动方式和依赖结构
- 根据这份需求生成一个前端页面原型
- 整理这个目录下的图片并按日期重命名
- 分析这份数据并生成报告

### 第三步：补充上下文

创建任务文档里给了几种方式：

- 用 `@` 引用文件、文档、规则
- 粘贴截图
- 上传文件
- 明确目标、范围、约束和输出格式

这一步非常关键。AI 工具用得顺不顺，很多时候不在于模型强不强，而在于上下文给没给够。你越清楚地告诉它输入、目标、输出格式和约束，它越不容易跑偏。

### 第四步：看结果区，不只看聊天区

CodeBuddy 的一个重要特点，是右侧结果区会展示产物、全部文件、变更和预览。也就是说，你不应该只盯着对话消息看，而是要去看它真正产出了什么文件、改了什么内容、结果是否可交付。

![CodeBuddy 创建任务页面截图](/posts-img/codebuddy-create-task-shot.jpg)

## CodeBuddy 的使用思路，别一上来就“全权托管”

很多人第一次用这类工具，最容易犯的错误就是：直接丢一句特别大的需求，然后等着它自己把全世界都做好。这样当然有可能成功，但也很容易翻车。

更推荐的方式，是分成三层来用。

第一层，让它做理解和计划。先问它怎么做，让它先拆任务。

第二层，让它做小步执行。比如先改一个模块、先做一个页面、先分析一个目录。

第三层，再逐步接入更复杂的工具和服务。

这个顺序的好处是非常稳。因为你能先知道它理解项目的方式对不对，再决定是否让它继续往下做。官方文档里虽然强调它能自主执行，但真正好用的前提，仍然是你要给它足够清晰的目标和边界。

## CodeBuddy 里怎么理解 MCP

虽然这篇不再展开“接入第三方 API”的具体路线，但 MCP 依然是理解 CodeBuddy 能力边界时绕不过去的一部分。

官方文档里对 MCP 的定义很明确：MCP 是一个开放标准，允许 CodeBuddy 与外部工具和数据源进行集成。它可以向模型提供上下文、暴露工具能力、构建可组合工作流，并通过本地运行方式更好地控制敏感数据。

你可以把它简单理解成：给 CodeBuddy 增加额外能力的一种标准接口。即使你暂时不用它，先知道它的存在和配置思路，也有助于你理解为什么 CodeBuddy 不只是一个“会聊天的插件”。

### MCP 支持哪些通信方式

官方文档提到主要支持三种传输方式：

- `stdio`：本地进程，通过标准输入输出通信
- `sse`：远程服务，通过 Server-Sent Events 通信
- `http`：远程服务，通过 HTTP 流式传输通信

### MCP 配置文件放哪

官方 CLI 文档给了明确说明。

用户级推荐配置文件：

- `~/.codebuddy/.mcp.json`

项目级推荐配置文件：

- `<项目根目录>/.mcp.json`

如果多个作用域里有同名服务，优先级是：

`local > project > user`

这个设计挺合理。全局常用工具放 user，项目专用服务放 project，临时实验型配置放 local，互相不打架。

![CodeBuddy MCP 文档页面截图](/posts-img/codebuddy-mcp-doc-shot.jpg)

### 最重要的安全建议

官方文档里反复强调一件事：敏感信息不要直接硬编码在配置文件里，尽量走环境变量展开。哪怕你只是先熟悉配置结构，也最好一开始就养成这个习惯。否则以后项目一多，回头收拾配置会很烦。

## CodeBuddy IDE 里怎么配置 MCP

如果你更习惯图形界面，而不是手工改配置文件，官方 IDE 文档里给了更直观的入口。

### 方式一：MCP Market 一键安装

在对话面板右上方进入 CodeBuddy Settings，切换到 MCP 标签页之后，你可以直接去 MCP Market 里一键安装已有 MCP Server。安装成功后会显示绿色状态，失败则显示红色状态。

这种方式适合：

- 你不想手改 JSON
- 你想快速试用现成服务
- 你需要低门槛接入已有生态

### 方式二：Add MCP 自定义配置

如果市场里没有你要的，或者你需要更细粒度地管理能力，就可以点 Add MCP，直接编辑 JSON 配置。

配置成功后，可以点击 Try to Run 按钮去验证是否可用，也可以直接在 Craft Agent 里输入任务，让 Agent 自动判断是否要调用 MCP 工具。

这个流程其实很重要，因为它说明 CodeBuddy 不只是“你手动选工具再调用”，而是可以在任务分析之后主动决定要不要用工具。也正因为如此，前面的权限控制和安全审批就特别关键。

## 用命令行管理 MCP，会更高效

如果你经常配置 MCP，手改 JSON 并不是效率最高的方式。CodeBuddy CLI 文档里给了比较完整的管理命令。

### 添加 stdio 类型服务

```bash
codebuddy mcp add --scope user my-tool -- /path/to/tool arg1 arg2
```

### 添加 HTTP 服务

```bash
codebuddy mcp add --scope project --transport http http-server https://example.com/mcp/http
```

### 添加 SSE 服务

```bash
codebuddy mcp add --scope user --transport sse sse-server https://example.com/mcp/sse
```

### 用 JSON 一次性添加

```bash
codebuddy mcp add-json --scope user http-server '{"type":"http","url":"https://example.com/mcp","headers":{"Authorization":"Bearer token"}}'
```

### 管理现有服务

```bash
codebuddy mcp list
codebuddy mcp get my-server
codebuddy mcp remove my-server
```

这些命令的价值在于，你可以把 MCP 配置管理做成真正的工程化流程，而不是靠手工点点点。尤其是项目级配置，配合版本控制，会非常适合团队协作。

## CodeBuddy Code 如何通过 API Key 和 CloudBase AI CLI 配置

除了桌面端和 IDE 使用方式，官方 CloudBase 文档还提供了一条很实用的方案：通过 CloudBase AI CLI 来配置 CodeBuddy Code。

这套方式支持两种登录方式：

- 使用 CodeBuddy 账号登录
- 使用 CodeBuddy API Key 配置

最基础的配置命令是：

```bash
tcb ai --setup
```

运行后，按向导选择：

1. 当前要配置的 AI 编程助手，选择 `CodeBuddy Code`
2. 选择配置方式
3. 完成账号登录或 API Key 配置
4. 设为默认助手（可选）

配置完成后，可以直接启动：

```bash
tcb ai -a codebuddy
```

如果已经设为默认值，那么直接运行：

```bash
tcb ai
```

也可以用环境变量方式：

```bash
export CODEBUDDY_API_KEY="your-api-key-here"
```

![CloudBase AI CLI 配置文档截图](/posts-img/codebuddy-cloudbase-shot.jpg)

这条路线很适合这几类人：

- 已经在用 CloudBase AI CLI
- 需要通过脚本或统一命令来管理多个 AI 助手
- 希望通过 API Key 做更明确的配置验证和权限管理

官方文档还提到，配置时会自动做 API Key 格式验证、网络连通性测试和 MCP 服务器配置检查，这对排错很有帮助。

## 实战里最容易踩的坑

### 1. 配置写了，结果没生效

这是最常见的。官方故障排除里也写得很直接：检查配置文件语法、确认作用域优先级、必要时重启 CodeBuddy 应用。

很多问题并不是配置错了，而是你以为它读的是 project，结果它先读了 user；或者你以为 local 生效了，结果当前会话压根没加载到。

### 2. 把 token 直接写进仓库

这个真别图省事。官方已经明确推荐使用环境变量，不要把 API Key、Authorization 头、认证信息硬写在配置里。尤其是团队项目，写进仓库等于给未来埋雷。

### 3. 一口气接太多工具

工具一多，上下文会变大，模型选工具也容易乱。官方给出的 `defer_loading` 就是为了解这个问题。能延迟加载的尽量延迟，不要啥都一股脑儿喂给模型。

### 4. 权限没收紧

MCP 工具权限支持 `deny`、`ask`、`allow` 三种规则，优先级也写得很清楚。最危险的不是没有能力，而是能力太多又没边界。像删除文件、写数据库、发请求这种工具，最好至少先走 `ask`，别上来就全放开。

### 5. 只会在图形界面点，不会用 CLI 管理

图形界面适合初学者，但只要你开始频繁管理 MCP，CLI 反而更稳、更快、更适合沉淀流程。能脚本化的就尽量脚本化，后面团队协作时你会感谢现在的自己。

## 一套适合新手的最小可用流程

如果你今天就想把腾讯 CodeBuddy 真正用起来，我建议按这个顺序来。

### 第一步：先从官方快速开始走通

装好工具，登录账号，创建第一个任务，熟悉左侧任务区、对话区和右侧结果区。先别急着扩展，先知道它原生工作流长什么样。

### 第二步：先用自然语言做一个技术任务

比如：

- 帮我梳理这个前端项目的启动方式和目录结构
- 帮我分析这个接口模块的调用关系
- 帮我把这段需求转成一个页面实现计划

### 第三步：先熟悉 MCP 的基础概念

不用急着上复杂配置，先知道配置文件在哪、作用域怎么选、验证怎么做。把这些基础概念搞明白，后面你再扩展能力时会顺很多。

### 第四步：把敏感信息改成环境变量

别偷懒，这一步越早做越好。不然以后再回头收拾配置，容易一地鸡毛。

### 第五步：用 Try to Run 或 CLI 验证配置

别上来就直接丢复杂任务。先确认工具能连、响应正常、权限合理，然后再逐步放进自己的日常流程。

### 第六步：再考虑 CloudBase AI CLI 和 API Key 管理

如果你要进入更稳定的工程化阶段，再把 `tcb ai --setup`、`CODEBUDDY_API_KEY` 这些配置加进来，让工具链更规范。

## 我的建议：把 CodeBuddy 当成“任务执行平台”，而不是“补全插件”

看完官方文档和社区文章后，我最大的感受是，很多人其实低估了 CodeBuddy。它当然能做代码补全、代码修复这些事情，但它真正有意思的地方，在于它已经不是一个单点能力，而是一个可以不断接入能力、不断扩展工作流的平台。

你越把它当“任务执行平台”来理解，越能把它用顺。

先让它理解任务，再让它动手；先从原生能力开始，再理解配置和能力扩展；先解决一个具体场景，再逐步把更多规则、更多工具、更多项目规范接进来。这样它才会越来越像你的长期协作者，而不是一个偶尔顺手用一下的 AI 插件。

如果你只是想体验，那当然装上就能开始。但如果你真想把它用进实际工作，我最推荐的路线还是这句老话：先上手，再标准化，再扩展能力。别一步冲太猛，不然人和工具一起翻车。
